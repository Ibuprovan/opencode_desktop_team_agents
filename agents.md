## 核心底层机制

在实施里程碑之前，全系 Agent 必须遵循以下三大底层工程铁律：

1. **以文件为唯一事实来源（Single Source of Truth）：** Agent 之间禁止纯上下文对话协作。所有交接物必须落盘（`docs/`、`docs/tasks/`）。
2. **Git 分支隔离机制：** 每个任务（TASK-xxx）启动时，系统底层自动创建 `feature/TASK-xxx` 分支，开发 Agent 仅在分支内编码。
3. **有限重试与仲裁机制：** 开发（`dev`）与审查（`reviewer`）在单个任务上的拉扯上限为 **3次**。超过3次未通过，任务挂起为 `BLOCKED`，自动唤醒 `architect` 或用户进行人工仲裁，切断 Token 死循环。

### 里程碑 1：核心 MVP 骨架（跑通流水线）

**目标：** 实现“需求输入 → 自动化分析 → 核心代码生成 → 实验总结”的最短闭环，满足基础课程设计与快速脚本开发。

#### 1. 激活 Agent 角色

- **name: pm (产品经理)**
- **职责：** 需求分析、用户场景设计、编写 PRD。
- **输出：** `docs/prd.md`
- **name: backend-dev (后端/核心算法工程师)**
- **职责：** 业务逻辑实现、核心算法与函数编写、API 实现。
- **输出：** `src/` 核心代码
- **name: report (报告生成器)**
- **职责：** 抽取 PRD 和代码结构，自动生成标准的实验报告、项目总结，处理 Markdown 格式。
- **输出：** `reports/experiment-report.md`

#### 2. 双轨工作流（本阶段默认走“快轨”）

用户输入指令 → `pm` 生成 PRD → `backend-dev` 编写单兵代码 → `report` 汇总输出。

#### 3. 交付标准（当前阶段产出）

- [ ] 建立初始化目录结构（`docs/`, `src/`, `reports/`）。
- [ ] 提供 `pm`、`backend-dev`、`report` 的 Prompt 定义配置。
- [ ] 演示一个简单的 Python 课程设计项目（如：经典算法实现或基础 Web 爬虫）从零生成到报告出炉的全过程。

> ⚠️ **Agent 暂停点：** 请向用户汇报里程碑 1 的目录结构、Agent 配置及测试运行结果。
> **等待用户指令：** “同意进入里程碑 2”

### 里程碑 2：任务驱动与质量门禁（工程化进阶）

**目标：** 引入任务系统（Tasks），打破长文本大模型的幻觉，并加入 `reviewer` 门禁，确保代码健壮性，防止代码越改越乱。

#### 1. 扩充 Agent 角色

- **name: pmo (项目管理办公室)**
- **职责：** 接收用户或 PM 需求，拆解任务，创建并更新 `docs/tasks/TASK-xxx.md`。
- **输出：** `docs/project-plan.md`，任务看板
- **name: architect (系统架构师)**
- **职责：** 技术选型、模块划分、接口规范（API Spec）定义、**技术仲裁**。
- **输出：** `docs/architecture.md`, `docs/api-spec.md`
- **name: reviewer (代码审查员)**
- **职责：** 依据规范检查代码，提供重构建议，控制代码合规性。
- **输出：** `docs/review-report.md`

#### 2. 任务流转状态机（Task State Machine）

所有 Agent 严格通过修改 `docs/tasks/TASK-xxx.md` 中的 `Status` 字段进行协作：

```
TODO → IN_PROGRESS → REVIEWS → (通过) → DONE
                     ↑
                (拒绝 ≤ 3 次)     ↓ (拒绝 > 3 次)
              backend-dev 修正    BLOCKED (唤醒 architect 仲裁)
```

#### 3. 交付标准

- [ ] 完善 `docs/tasks/` 目录规范，给出标准 `TASK.md` 状态机模板。
- [ ] 提供 `pmo`, `architect`, `reviewer` 的 Agent 角色实现。
- [ ] 演示在“拉扯/修改”场景下，`reviewer` 拒绝低质量代码、促使 `backend-dev` 修复并最终合并的流程。

> ⚠️ **Agent 暂停点：** 请向用户汇报里程碑 2 的状态机代码逻辑、任务模版及双 Agent 纠错对抗测试记录。
> **等待用户指令：** “同意进入里程碑 3”

### 里程碑 3：长期记忆与全量交付（全栈系统开发）

**目标：** 解决多文件、跨周期的“长效开发”问题。引入全栈开发能力及基础设施层的记忆代理，实现完整项目的全量交付。

#### 1. 扩充 Agent 角色

- **name: memory (记忆中台)**
- **职责：** 维护 `project-memory.md`。每次任务关闭时更新拓扑和决策。执行新任务前，动态提取相关上下文片段，拒绝“全量盲读”以节省 Token。
- **输出：** 动态上下文快照、`project-memory.md`
- **name: frontend-dev (前端工程师)**
- **职责：** Web 前端、页面 UI、状态管理与接口对接。
- **name: devops (交付工程师)**
- **职责：** Dockerfile 编写、CI/CD 脚本、一键部署配置（`deployment/`）。
- **输出：** `deployment/docker-compose.yml`, `docs/deployment.md`

#### 2. 精细化记忆管理策略

`project-memory.md` 严禁记录冗长代码，仅允许更新：

- 当前系统拓扑结构与技术栈。
- 核心数据库 Schema / 实体关系。
- 已完成的 TASK 关键决策索引。

#### 3. 交付标准

- [ ] 交付 `memory` 智能体的增量写入与检索规则。
- [ ] 实现全栈前后端分离项目（如：带 JWT 鉴权的套件管理系统）的联合开发闭环。
- [ ] 生成完整的交付物包：代码 + 部署脚本 + 用户手册（`docs/user-guide.md`）。

> ⚠️ **Agent 暂停点：** 请向用户汇报里程碑 3 的记忆更新算法、全栈协作链路以及 DevOps 自动生成的部署配置文件。
> **等待用户指令：** “同意进入里程碑 4”

### 里程碑 4：网安与自治特化（高级工具与审计）

**目标：** 面向网络安全项目进行深度定制，使系统具备漏洞挖掘、自动化代码审计与渗透测试辅助的自治能力。

#### 1. 扩充 Agent 角色

- **name: security (合规安全官)**
- **职责：** 在代码合并进主干前进行 OWASP 风险评估、权限检查，阻断高危代码。
- **输出：** `docs/security-report.md`
- **name: cyber (网安特化智能体)**
- **职责：** 独立运行静态代码审计（SAST）、构造漏洞 PoC 验证脚本、辅助渗透测试分析。
- **输出：** `reports/vulnerability-assessment.md`
- **name: research (前沿调研员)**
- **职责：** 针对特定漏洞 CVE、开源仓库或学术论文进行快速 Read 与深度调研，提供技术攻关报告。
- **输出：** `docs/research.md`

#### 2. 安全工程全流转质量门

代码生成 → `reviewer` 规范检查 → `security` 安全漏扫（拒绝 SQL 注入、硬编码密钥、未授权访问等） → 允许合入主干。

#### 3. 交付标准

- [ ] 交付 `security` 和 `cyber` Agent 的专属漏洞扫描与审计 Prompt 集。
- [ ] 模拟实战场景：对一个含有经典漏洞（如 Log4j2 或 Web 鉴权绕过）的代码仓库，`cyber` 成功定位漏洞、`research` 提供修复思路、`backend-dev` 修复、`security` 审计合规的全自治过程。
